## 1. 開發流程

```
CEO 確認需求 → 開 Work Order → 小c實作 → 自驗 → Code Review → QA → Deploy
```

| 階段 | 負責人 | 產出 |
|:-----|:------|:-----|
| 需求 | CEO | Work Order（WO#00X） |
| 實作 | 小c | 功能程式碼 + Unit Test |
| 自驗 | 小c | 驗證結果（至少 3 項檢查通過） |
| Code Review | CEO | PR Review Comment |
| QA | 小陳（暫代） | Test Report |
| Deploy | CEO | SSH Pi4 部署 + Health Check |

## 2. Branch 規範

| Branch | 用途 | 基於 | Merge 目標 |
|:-------|:-----|:-----|:-----------|
| `main` | 正式上線 | — | — |
| `dev` | 開發整合 | `main` | `main` |
| `fix/*` | Bug 修復 | `dev` 或 `main` | `dev` |
| `feat/*` | 新功能 | `dev` | `dev` |

**規則：**
- 開發前先開 Branch，不在 `main` 上直接改
- Branch 命名：`fix/<簡短說明>` 或 `feat/<簡短說明>`
- 完成後開 PR → Code Review → Merge

## 3. Code Review 檢查清單

小c（Coder）交 PR 前必須先自檢以下項目，CEO Review 時逐一驗證：

### 3.1 Import 檢查
- [ ] 所有新引入的 class/function 都有對應的 `import` 或 `from ... import`
- [ ] 執行 `python3 -c "from 你的新模組 import 你的新Class"` 確認無誤

### 3.2 設定檔檢查
- [ ] 新增的 config 變數有在正確的 `.env` 檔案中
- [ ] 確認 config 讀取路徑（`config.py` 中 `env_path` 指向哪個檔案）
- [ ] ConfigParser section 名稱（`[V1]`/`[V2]`/`[V3]`）正確
- [ ] key 名稱大小寫與 `field_map` 一致

### 3.3 基本驗證
- [ ] 服務能正常啟動（無 import error、無 syntax error）
- [ ] Endpoint 回傳預期 status code
- [ ] 至少執行一次 E2E 測試（上傳檔案 → 查詢 → 確認結果）

### 3.4 異常處理
- [ ] 外部 API 呼叫有 try/except 和重試機制
- [ ] 錯誤訊息包含足夠的診斷資訊
- [ ] 不直接噴出原始 exception 給前端

### 3.5 Frontend 驗證（QA 必測）
- [ ] 修改 upload zone accept / hint 後，**用手機瀏覽器實際操作一次**
- [ ] 選取圖片（png/jpg 各一）確認可上傳、顯示 `status: indexed`
- [ ] 選取不支援格式確認顯示正確 error message
- [ ] 手機上傳按鈕無 layout 破版（Android Chrome）
- [ ] 清除對話按鈕功能正常
- [ ] Demo 頁面載入無 console error

## 4. 開發注意事項（已知陷阱）

### 4.1 Config / .env 路徑

**`backend/app/config.py`** 的 `get_settings()` 讀取的是：

```
os.path.join(os.path.dirname(__file__), "..", "..", ".env")
```

即：**專案根目錄的 `.env`**，而非 `backend/.env` 或其他目錄。

```
cram_service/
├── .env                    ← ⚠️ 就是這個，不是別的
├── backend/
│   ├── .env                ← ❌ 讀不到，別寫這裡
│   ├── app/
│   │   └── config.py       ← 從這裡推算路徑
```

**新增 config 變數時：**
1. 在 `backend/app/config.py` 的 `Settings` class 新增欄位
2. 在 `field_map` 新增 mapping（key 名稱大寫）
3. 在根目錄 `.env` 的 `[V1]` section 寫入值

### 4.2 INI Section Config 格式

.env 使用 configparser INI 格式，不是標準的 key=value 格式：

```
[DEFAULT]
APP_CONFIG_VERSION = V1

[V1]
GOOGLE_VISION_API_KEY=xxx   ← 大寫 key，在 [V1] 下面
```

- `APP_CONFIG_VERSION` 決定讀哪個 section（`V1`/`V2`/`V3`）
- 新增的 key 必須放在**對應的 section** 裡面
- 大小寫不重要（code 會自動 `k.upper()`），但建議統一用大寫

### 4.3 Import 缺失（最常見錯誤）

在 `documents.py` 新開了一個 class/service，但在 **import 區塊** 忘了加 import：

```python
# ❌ 錯誤：用了 OCRService 但沒 import
ocr = OCRService()

# ✅ 正確：先在檔案頂部加入
from app.services.ocr import OCRService
```

**預防方法**：寫完新檔案後，馬上執行：

```bash
cd ~/cram_service
PYTHONPATH=backend .venv/bin/python3 -c "from app.services.ocr import OCRService"
PYTHONPATH=backend .venv/bin/python3 -c "from app.routers.documents import *"
```

### 4.4 Python Path

Pi4 的 service 透過 systemd 啟動，`WorkingDirectory=/home/apollo/cram_service`，且有設 `PYTHONPATH`。手動測試時：

```bash
cd ~/cram_service
PYTHONPATH=backend .venv/bin/python3 -c "你的測試"
```

## 5. 部署 SOP

```bash
# 1. 確認 branch 正確
git branch                  # 確認在正確 branch

# 2. Commit & Push
git add -A
git commit -m "feat: 簡短說明"
git pull --rebase origin <branch>   # 先 rebase 遠端
git push origin <branch>

# 3. 重啟服務
systemctl --user restart cramai.service

# 4. Health Check
sleep 5
curl -s http://localhost:8000/health
curl -s -o /dev/null -w "%{http_code}" http://localhost:8000/demo

# 5. E2E Test（至少上傳一個測試檔案並查詢）
curl -s -X POST http://localhost:8000/api/v1/tenants/default/documents/upload \
  -F "file=@/tmp/test-file.pdf"
```

## 6. Work Order 規範

每次開發任務必須開立 **Work Order**，記錄於 `docs/work-orders/WO###.md`：

| 欄位 | 說明 |
|:-----|:-----|
| 修改原因 | 背景、問題、痛點 |
| 修改結果 | 預期完成後長怎樣 |
| Branch | 開發分支名稱 |
| 修改檔案 | 檔案列表 + 操作類型（新增/修改） |
| 驗收項目 | 逐項列出驗收方式 |
| 執行者 | 誰開發、誰 review |
