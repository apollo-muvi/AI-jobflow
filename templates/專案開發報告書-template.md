## 三、專案開發指導原則

### 3.1 開發流程

```
CEO 確認需求 → 開 Work Order → Coder 實作 → 自驗 → Code Review → QA → Deploy
```

| 階段 | 負責人 | 產出 |
|:-----|:------|:-----|
| 需求 | CEO | Work Order（WO#00X） |
| 實作 | | |
| 自驗 | | |
| Code Review | | |
| QA | | |
| Deploy | | |

### 3.2 Branch 策略

| Branch | 用途 | 基於 | Merge 目標 |
|:-------|:-----|:-----|:-----------|
| `main` | 正式上線 | — | — |
| `dev` | 開發整合 | `main` | `main` |
| `fix/*` | Bug 修復 | `dev` 或 `main` | `dev` |
| `feat/*` | 新功能 | `dev` | `dev` |

### 3.3 版號規則

```
0.1 → 0.2 → 0.3 → ...（開發階段，每完成一個 task +0.1）
1.0 → 1.1 → 1.2 → ...（Release 階段，CEO 決定）
```

### 3.4 Code Review 檢查項目

- [ ] Import 正確性
- [ ] Config/.env 路徑正確
- [ ] 服務正常啟動
- [ ] Endpoint 回傳正確 status code
- [ ] 至少一次 E2E 測試
- [ ] 外部 API 有 try/except + 重試
- [ ] 手機端 UI 無 layout 破版（如有前端）

### 3.5 部署 SOP

```bash
# 1. 確認 branch
# 2. Commit & Push
# 3. 重啟服務
# 4. Health Check
# 5. E2E Test
```
